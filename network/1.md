### tcp对比udp

```
TCP UDP是传输层

TCP有状态有连接和断开，稳定传输
TCP有流量控制，通信双方各声明一个窗口标识自己当前的处理能力
UDP无连接和断开，不稳定传输但效率高
UDP没有拥塞控制，网络出现拥塞不会使源主机的发送速率降低
```



### http各版本对比

```
2.0 二进制分帧，压缩header，减少请求传输的文件体积，多路复用，一次tcp连接中可以多个http并行请求，服务端推送 
1.1 缓存策略，长连接connection：keep-alive 一次tcp连接多次请求，大文件断点续传 状态码206，支持新的请求方法put delete可用于restful api
1.0 最基础的http协议，支持基本的get和post方法，目前已没人使用

  http2只解决了http层面（应用层）的队头阻塞问题：http1.1同时打开多个tcp连接但是单个连接依然会产生阻塞即使后续请求不依赖于当前阻塞的请求，http2引入多路复用解决了这个问题，允许在单个tcp连接上发送多个请求和响应
  
  http2没解决tcp层面的队头阻塞问题：TCP是一个面向连接的、可靠的、基于字节流的传输层通信协议，它保证数据包的顺序传输和可靠传输。TCP使用顺序编号来确保数据包的顺序性，如果一个数据包丢失或延迟，TCP会等待这个数据包被重传，而在这个等待期间，即使有其他数据包已经准备好发送，它们也不能被发送，直到丢失的数据包被成功传输。这种机制导致了TCP层面的队头阻塞。
  
  HTTP/3通过基于UDP的QUIC协议，旨在解决TCP层面的队头阻塞问题，因为QUIC提供了无需等待即可继续发送数据包的能力，从而减少了丢包带来的影响。
  
```



### https

```
服务端响应请求时携带数字证书（包含公钥A），客户端取出公钥，用公钥加密随机码，将随机码发送给服务器，服务端使用私钥解密（至此为非对称加密）。将解密后的随机码作为密钥进行后续对称加密

中间人攻击就是黑客伪造服务端公钥进行和服务端相同的操作（可以通过正规的数字证书防止该攻击）
```

### get和post的区别

```
数据传输方式：

GET 请求会将参数附加在URL之后，通过URL传递数据。参数通过查询字符串（query string）传递，例如：http://example.com/api?param1=value1&param2=value2。
POST 请求则是将数据放在请求体（request body）中，通过这种方式可以传输比GET更多的数据。

幂等性：

GET 请求是幂等的，意味着多次执行相同的GET请求，资源的状态不会改变，这通常用于数据的读取。
POST 请求不是幂等的，多次发送相同的POST请求可能会每次都对服务器产生不同的副作用，如创建多个资源。

缓存：

GET 请求可以被浏览器缓存，这意味着对相同URL的重复请求可能会直接从浏览器的缓存中读取数据，而不需要再次请求服务器。
POST 请求通常不会被缓存，每次发送都需要服务器的响应。

数据大小：

GET 请求由于参数在URL中，受到URL长度限制，因此传递的数据大小有限。
POST 请求没有这个限制，可以传输更大的数据。
请求行长度限制：

一些服务器可能会限制请求行的长度，这可能会影响到GET请求的参数长度，而POST请求则不会受到这个限制。

用途：

GET 请求通常用于请求服务器发送资源，比如获取网页或搜索结果。
POST 请求常用于触发服务器上的操作，如表单提交，创建资源等。

编码类型：

GET 请求的参数通常使用URL编码。
POST 请求的参数可以是URL编码，也可以是其他类型，如JSON、XML等，这取决于Content-Type请求头的设置。
```
